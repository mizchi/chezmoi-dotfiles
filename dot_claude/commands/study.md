対象の公開面（パブリック API / エクスポート）をレクチャー → 理解度チェック → 弱点補強のループで、ユーザーが「何ができるか」を把握できているか確認する。

ユーザーが AI 生成コードや外部ライブラリの中身を読まないまま使い続けることへの対策として手動起動するコマンド。普段は発動しない。

## 引数

`$ARGUMENTS` — 学習対象。次のいずれか:
- ローカルパス（`src/parser/`、`./packages/foo/index.ts`）
- インストール済みパッケージ名（`zod`、`hono`）
- ライブラリ名 + バージョン（`zod@4`）

省略された場合、何を学びたいか 1 度だけ質問する。

## 前提

このコマンドは **対話必須**。auto モード下でも、各フェーズの終わりにユーザーの応答を待ってから次へ進む。一気に全フェーズを出力しない。

## フェーズ 1: 公開面の特定

対象に応じて public surface を抽出する。internal / private / `_` prefix / `@internal` は除外。

- **ローカルパス**: `package.json` の `main`/`module`/`exports`/`types` を起点に export を列挙。`moon.pkg.json` なら `"export"` フィールド。
- **node_modules パッケージ**: `node_modules/<name>/package.json` の `exports` と `*.d.ts` から API を抽出。
- **ライブラリ名のみ（未インストール）**: 公式ドキュメントまたは型定義パッケージから抽出。曖昧なら何を見たかを明記。
- **単一ファイル**: そのファイルの export だけに絞る。

抽出結果（API 名、シグネチャ、出典）を内部メモとして保持。

## フェーズ 2: レクチャー

抽出した API を 3〜7 個の **責務単位のグループ** に整理し、グループごとに次の形式で提示する:

```
### <グループ名>

- `<シグネチャ>` — 何をするか（1 行）／使う場面（1 行）
- ...
```

ルール:
- 識別子・シグネチャは原文のまま（翻訳しない）
- 「〜について説明します」のような前置きは書かない
- 全体を 30〜80 行に収める。冗長になったらグループを統合する
- アルファベット順ではなく **ユーザーが最初に触るもの順** に並べる

末尾に「フェーズ 1 についてここまでで質問はあるか？」と尋ねて応答を待つ。

## フェーズ 3: 理解度チェック

3〜5 問のクイズを出題。次のタイプから 2 種類以上を混ぜる:

- **用途選択**: 「X したいときどの API を使うか」
- **シグネチャ読解**: 「`f(a, b)` で `a` に Y を渡したら何が返るか / どう型推論されるか」
- **比較**: 「A と B はどう違うか / どちらが正しいか」
- **誤用検出**: 「次のうち間違っているのはどれか、なぜ」

出題ルール:
- 1 問ずつではなく全問まとめて提示
- 選択式と記述式を混ぜる
- すべての回答が揃うまで採点しない（部分回答でも待つ）
- 公開面の中で **使用頻度が高いもの** を優先して問う

ユーザーの回答を受けて採点し、各問について「正解 / 不正解 / 曖昧」を明記。

## フェーズ 4: 補強ループ

不正解・曖昧だった箇所のみ:

1. その API を再説明する（具体例コード 1 つ含める、3〜10 行）
2. 同じトピックで質問を 1〜2 問追加出題
3. 回答を待つ → 採点

全問正解 or ユーザーが「ストップ」と書く or 3 ループ到達で終了。

終了時、次を 5 行以内で要約:
- 把握できている API 群
- まだ曖昧な API 群（あれば）
- 推奨する次のアクション（実コードを書いて試す / 公式ドキュメント該当節を読む 等）

## 出力ルール

- 日本語
- コード識別子・シグネチャは原文のまま
- 1 フェーズの出力ごとに必ずユーザー応答を待つ
- レクチャー・解説は密度優先。冗長な前置き禁止
